- **Fecha:** 2026-05-04
- **Tarea backlog:** `P2 | sdk | Plan de regen tygo: actualizar tygo.yaml y metacore-sdk/packages/sdk/src/types/manifest.ts para reflejar GAP-1, GAP-2, GAP-3 una vez mergeados en kernel. Solo plan + Changeset draft.`
- **Alcance:** documento de plan + changeset draft. **No se ejecuta** `pnpm codegen` todavía: los GAPs aún no están mergeados en `metacore-kernel/manifest`. Esta entrega prepara el terreno para que, en el momento en que el kernel publique las struct nuevas, el regen sea mecánico.

### Estado actual del SDK

- `tygo.yaml` (root del monorepo) define un único package mapping:
  - source: `github.com/asteby/metacore-kernel/manifest`
  - output: `packages/sdk/src/generated/manifest.ts`
  - mappings: `time.Time → string`, `json.RawMessage → unknown`.
  - frontmatter custom recordando regenerar con `pnpm codegen`.
- El path real del archivo TS generado es `packages/sdk/src/generated/manifest.ts`. **El backlog cita `packages/sdk/src/types/manifest.ts`, que no existe**: la convención del repo es que el archivo público se llama `types.ts` (handwritten, re-exporta los símbolos generados) y `generated/manifest.ts` es el output literal de tygo. Se documenta abajo cómo resolver esa ambigüedad sin renombrar nada.
- `packages/sdk/src/types.ts` re-exporta nominalmente cada símbolo generado; cualquier struct nuevo (`ValidationRule`, `RelationDef`, `TriggerDef`) debe agregarse a esa lista o no quedará accesible para los consumers.
- `packages/sdk/package.json` versión actual: **`@asteby/metacore-sdk@2.3.0`**. Cualquier bump de los tipos generados → changeset minor (campos opcionales nuevos, no breaking).

### Disonancia de path del backlog

El backlog dice `metacore-sdk/packages/sdk/src/types/manifest.ts` (carpeta `types/`). En el repo no existe esa carpeta; existen:

- `packages/sdk/src/types.ts` (handwritten, re-exporta)
- `packages/sdk/src/generated/manifest.ts` (output literal de tygo, regenerado)

**Decisión:** mantener la estructura actual. NO crear `src/types/manifest.ts` ni mover archivos: el código generado vive bajo `generated/` por convención (todos los demás packages lo respetan) y `types.ts` es el barrel handwritten. Cuando ejecutemos el regen, modificaremos `generated/manifest.ts` (auto) y editaremos `types.ts` para re-exportar los símbolos nuevos.

## Plan de regen — pasos en orden

### Pre-requisitos (en kernel, fuera del scope de esta tarea)

Antes de correr `pnpm codegen` en el SDK, los siguientes commits deben estar mergeados en `metacore-kernel:main` (Fase 1 del backlog):

1. GAP-1 implementado: campos `Visibility`, `Searchable`, `Validation`, `Widget` agregados a `manifest.ColumnDef` + struct `ValidationRule` exportado. Tests verdes. CHANGELOG minor.
2. GAP-2 implementado: struct `RelationDef` exportado + `ModelDefinition.Relations []RelationDef`. Validate verifica `Kind` válido y `Through` requerido si `Kind="many_to_many"`. Tests verdes. CHANGELOG minor.
3. GAP-3 implementado: struct `TriggerDef` exportado + `ActionDef.Trigger *TriggerDef`. Validate exige `Function ∈ Backend.Exports` cuando `Type="wasm"`. Tests verdes. CHANGELOG minor.
4. `metacore-kernel` publica un tag (sugerido `v0.X.0`) que el SDK pueda fijar en `replace`/`go.work` durante el codegen.

> **No** vamos a esperar al bump de `APIVersion`: GAP-1/2/3 se diseñaron como aditivos no breaking. `APIVersion` queda en `"2.0.0"` hasta que llegue un breaking. Si el kernel decide bumpearlo, el regen lo refleja automáticamente.

### Paso 1 — actualizar `tygo.yaml`

El mapping actual cubre `github.com/asteby/metacore-kernel/manifest` y emite a un único output. Los structs nuevos (`ValidationRule`, `RelationDef`, `TriggerDef`) están en el **mismo package** Go (`manifest`), así que **NO** hace falta agregar más entries en `packages:`. El único cambio necesario es asegurar que las type mappings siguen cubriendo cualquier tipo Go nuevo que aparezca.

Revisión de tipos esperados en los GAPs:

| Tipo Go esperado | Mapping necesario | Comentario |
|------------------|-------------------|-----------|
| `string`, `bool`, `int` (campos primitivos) | n/a (default) | tygo los emite como `string`, `boolean`, `number` directos. |
| `*ValidationRule` (puntero a struct) | n/a | tygo emite `validation?: ValidationRule;` automáticamente. |
| `[]RelationDef` | n/a | array de struct local. |
| `map[string]string` (ej. `ErrorMap`) | n/a | ya hay precedente (`HookDef` map similar). |
| `time.Time` | ya mapeado a `string` | sin cambios. |
| `json.RawMessage` | ya mapeado a `unknown` | sin cambios. |
| `uuid.UUID` (no esperado en GAPs) | si aparece, agregar mapping → `string` | **VERIFICAR** al correr el codegen — si falla, agregar línea. |

**Cambio propuesto en `tygo.yaml`:** añadir `"uuid.UUID": "string"` y `"sql.NullString": "string | null"` como **defensivo** (no hay evidencia hoy de que se usen, pero los GAPs nuevos podrían introducirlos al validar). Si al correr `pnpm codegen` aparecen `unknown` raros, incorporar el mapping faltante en este mismo archivo.

```yaml
packages:
  - path: "github.com/asteby/metacore-kernel/manifest"
    output_path: "packages/sdk/src/generated/manifest.ts"
    type_mappings:
      "time.Time": "string"
      "json.RawMessage": "unknown"
      # Defensivos para GAP-1/2/3 — confirmar al primer run:
      "uuid.UUID": "string"
      "sql.NullString": "string | null"
    frontmatter: |
      /**
       * AUTO-GENERATED — do not edit by hand.
       * Re-generate with: pnpm codegen
       * Source: github.com/asteby/metacore-kernel/manifest
       */
```

> Si tras el primer `pnpm codegen` los tipos generados no usan `uuid.UUID` ni `sql.NullString`, **revertir** esas dos líneas para mantener el archivo mínimo. Mantener el YAML lo más cercano posible al estado actual.

### Paso 2 — correr `pnpm codegen`

Desde la raíz del monorepo:

```bash
pnpm codegen
```

Esto invoca `go run github.com/gzuidhof/tygo@v0.2.17 generate` y reescribe `packages/sdk/src/generated/manifest.ts`. La salida esperada incluirá:

- `interface ColumnDef` con 4 campos opcionales nuevos: `visibility?`, `searchable?`, `validation?`, `widget?`.
- `interface ValidationRule` (nuevo).
- `interface RelationDef` (nuevo).
- `interface ModelDefinition` con campo opcional `relations?: RelationDef[]`.
- `interface TriggerDef` (nuevo) y `interface ActionDef` con `trigger?: TriggerDef`.
- Posibles cambios en comentarios doc.go del kernel — propagados literalmente como JSDoc por tygo.

**Validación post-regen:**

```bash
pnpm typecheck   # turbo run typecheck en todos los packages
pnpm test        # smoke a tests del SDK
```

`runtime-react`, `auth`, `app-providers` no deberían romperse: los campos son opcionales y nadie los lee aún.

## Riesgos y mitigaciones

| Riesgo | Mitigación |
|--------|------------|
| Kernel mergea GAP-1/2/3 con naming distinto al esperado (`Validation` vs `ValidationRule`, `Trigger` vs `TriggerDef`) | Tygo es declarativo: el regen reflejará el naming real del Go. El re-export de `types.ts` se ajusta a lo que efectivamente se emite. **No** acoplar este plan a un naming específico — usar el output del primer `pnpm codegen` como autoridad. |
| Aparece un tipo Go nuevo (`time.Duration`, `pq.StringArray`) sin mapping en `tygo.yaml` | Tygo lo emite como `unknown`. Si typecheck pasa, dejarlo. Si no, agregar mapping puntual en este mismo archivo. |
| `pnpm codegen` falla por dependencia Go no resuelta (`go.work` apuntando a kernel local) | El monorepo ya tiene `go.work` que linkea kernel. Si el path local no apunta al kernel con los GAPs mergeados, el regen sale viejo. **Verificar** `cat go.work` antes de correr el codegen y, si es necesario, cambiar la ref a un tag publicado. |
| Consumers de `runtime-react` rompen porque ahora `ColumnDef.visibility` puede venir undefined o `"all"` y el código asume no existe | Falso positivo: ya hoy el campo no existe; agregarlo opcional **no** rompe el código que no lo lee. Las tareas Fase 2 explícitamente introducen el consumo. |
| Bump de versión `@asteby/metacore-sdk` rompe lockfiles de apps con `file:` deps (link, ops) | El changeset es minor, pnpm respeta el rango. Si alguna app usa `"workspace:*"` o `"file:"` literal, hay que correr `pnpm install` post-merge. Documentado en memoria `reference_link_deploy_sdk_sync.md`. |
