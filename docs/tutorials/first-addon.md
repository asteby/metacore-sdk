# Your first addon — `notes`

A walking-through tutorial that builds a real metacore addon end-to-end:
data model, sandboxed WASM backend, federated frontend, hot-swap. By the
last step you have a working sticky-notes panel that runs inside any
metacore host — and you understand every piece of the contract.

This tutorial assumes you read the diagrams in
[`docs/manifest-spec.md`](../manifest-spec.md) and
[`docs/wasm-abi.md`](../wasm-abi.md) at some point; it does not require
you to remember them.

## What we build

An addon `notes` that contributes:

- One model — `Note` (`title`, `body`, `pinned`) with the kernel's
  generated CRUD endpoints under `/dynamic/notes_note`.
- One sidebar entry that opens a federated React page rendering a grid of
  notes with create / edit / delete.
- One WASM action `auto_title` that reads a note's `body` server-side and
  rewrites the `title` from its first 60 characters. Runs inside the same
  DB transaction as the row update, so a rollback in either half rolls
  back the other.

The whole addon is under 200 lines of source. The shape mirrors the two
pilots already in production — `ops/addons-wasm/fiscal_mexico` (single
model + WASM action) and `ops/addons-wasm/kitchen-display` (federated
remote + immersive layout). Both are linked at the bottom as the next
step after this tutorial.

## Prerequisites

- A metacore host already running locally — `ops` (or any app on the
  starter) works. The host must expose `/api/dynamic/...` (kernel
  `dynamic` package, mounted by default) and load federated remotes via
  `<AddonLoader>` from `@asteby/metacore-runtime-react`.
- Go `1.25+` with `wasip1` target available (the stock toolchain — no
  TinyGo required).
- Node `22+` and pnpm `10+` for the frontend remote.

## What this tutorial does **not** cover

- Signing the addon for marketplace publish — `signature.json` is left
  with `verified: false` and the dev-mode `ALLOW_UNSIGNED_BUNDLES` flag.
  The full signing path lives in [`docs/PUBLISHING.md`](../PUBLISHING.md).
- `tenant_isolation: "schema-per-tenant"` — we use the default `"shared"`
  (one schema, RLS by `organization_id`). The regulated-data layout is a
  follow-up tutorial.
- Declaring `http:fetch`, `event:subscribe` or any capability beyond DB
  read/write. See [`docs/capabilities.md`](../capabilities.md).
- LLM tools (`manifest.tools`). They're optional; the notes addon does
  not register one.

## Step 1 — The structure of an addon

Every metacore addon is a directory the kernel can ingest as a bundle.
The layout below is the minimum that compiles, signs and installs:

```
notes/
├── manifest.json          # the contract — model, frontend, backend
├── backend/
│   ├── go.mod             # standalone Go module (GOWORK=off)
│   ├── main.go            # WASM exports following the kernel ABI
│   └── build.sh           # wasip1 reactor build + signature.json
├── frontend/
│   ├── package.json
│   ├── tsconfig.json
│   ├── vite.config.ts     # federation remote config
│   ├── index.html
│   └── src/App.tsx        # the exposed React page
├── signature.json         # placeholder until the marketplace re-signs
└── README.md              # what this addon does (optional)
```

Create it now:

```bash
mkdir -p notes/backend notes/frontend/src
cd notes
touch manifest.json signature.json README.md
touch backend/go.mod backend/main.go backend/build.sh
chmod +x backend/build.sh
touch frontend/package.json frontend/tsconfig.json frontend/vite.config.ts frontend/index.html
touch frontend/src/App.tsx
```

The file contents are filled in over the next four steps.

## Step 2 — `manifest.json`

This is the only file the kernel **requires**. Everything else (a WASM
module, a federated remote, settings) is opt-in. The schema lives in
`kernel/manifest/manifest.go`; the TS mirror is generated by `tygo` and
shipped as `@asteby/metacore-sdk` types.

Paste the file below into `notes/manifest.json`:

```json

  "name": "Notes",

  "version": "0.1.0",

  "icon": "StickyNote",

    "entry":     "/api/metacore/addons/notes/frontend/remoteEntry.js",

    "layout":    "shell"
  },

  "backend": {
    "runtime":         "wasm",
    "entry":           "backend/main.wasm",
    "exports":         ["alloc", "free", "auto_title"],
    "memory_limit_mb": 32,
    "timeout_ms":      3000
  },

  "capabilities": [
    { "kind": "db:read",  "target": "addon_notes.*", "reason": "Read user notes" },
    { "kind": "db:write", "target": "addon_notes.*", "reason": "Persist notes + auto-title" }
  ]
}
```

What each block does, with a pointer back to the Go source so you can
read the validation rules:

- `key`, `name`, `version`, `kernel` — identity and compat window. The
  kernel rejects any installed addon whose host version doesn't satisfy
  the range. See `manifest.go:9-26`.
- `tenant_isolation: "shared"` — single schema `addon_notes`,
  RLS-scoped by `organization_id`. The other supported value is
  `"schema-per-tenant"` for regulated data. See `manifest.go:31-40`.
- `model_definitions[]` — each entry creates `addon_notes.note` at
  install time. The `model_key` (`notes_note`) is what every API and SDK
  call addresses the model by. The column shape matches what the two
  pilot addons use (`ops/addons-wasm/fiscal_mexico/manifest.json:13-30`,
  `ops/addons-wasm/kitchen-display/manifest.json:34-52`). `searchable`,
  `widget` and the `default` value are part of the v0.10 column
  metadata extension (kernel PR #45, GAP-1).
- `navigation[]` — a sidebar group that opens the model's auto-generated
  list page. With `model` set, the URL `addon://notes/` is resolved by
  the runtime to the dynamic CRUD route bound to `notes_note`.
- `actions.notes_note[]` — one per-row action. `trigger.type: "wasm"`
  tells the kernel to dispatch to `Backend.Exports["auto_title"]`
  instead of a webhook; `run_in_tx: true` keeps the export call inside
  the same DB transaction as the row mutation that prompted the action,
  so a failure rolls everything back. The trigger contract is at
  `manifest.go:252-284`.
- `frontend.layout: "shell"` — the host wraps our component in the
  default chrome (sidebar + topbar). Switch to `"immersive"` in Step 8
  to see the full-page mode kitchen-display uses
  (`manifest.go:125-141`).
- `backend.exports` must enumerate every symbol the host can dispatch
  to. `alloc` and `free` are the ABI primitives (see Step 3); we add
  `auto_title` for our action. See `manifest.go:151-158`.
- `capabilities[]` — declared scopes the runtime enforces. We only need
  read/write on our own schema. Adding e.g. `{"kind":"http:fetch",
  "target":"https://api.example.com/*"}` would route through the SSRF
  guard. See `docs/capabilities.md`.

## Step 3 — Backend WASM

The kernel runs addon backends inside a wazero sandbox configured with
the **metacore ABI v1** (mirrored at `kernel/runtime/wasm/abi.go:10-26`
and on the frontend side at `packages/sdk/src/wasm-client.ts:19-43`).
The contract is tiny:

- The guest exports `memory`, `alloc(size: i32) -> i32` and
  `free(ptr: i32, size: i32)`.
- Each handler is `(ptr: i32, len: i32) -> i64` where the return packs
  pointer and length: `(uint64(ptr) << 32) | uint64(len)`.
- A return of `0` means "empty success". An error envelope is just a
  JSON `{"success": false, "error": {...}}` written into the returned
  slot.

### `backend/go.mod`

The backend lives in its own Go module so its dependency closure stays
minimal and reproducible. It does **not** depend on the kernel.

```text
module github.com/you/notes/backend

go 1.25
```

### `backend/main.go`

```go
// Package main is the WASM module for the `notes` addon.
//
// Build:
//   GOWORK=off GOOS=wasip1 GOARCH=wasm \
//       go build -buildmode=c-shared -trimpath -o main.wasm .
//
// The module exposes the kernel ABI v1 contract documented in
// metacore-kernel/runtime/wasm/abi.go:
//
//   - `memory`                    (default export, provided by the runtime)
//   - `alloc(size i32) i32`       bump allocator, returns guest pointer
//   - `free(ptr i32, size i32)`   no-op for a bump allocator
//   - `auto_title(ptr, len) i64`  action handler, returns (ptr<<32)|len
package main

import (
	"encoding/json"
	"strings"
	"unsafe"
)

// Bump allocator state. The host writes the request payload at the
// address `alloc` returned and calls `auto_title(ptr, len)` next.
// We start above page-zero so any host-side off-by-one is loud.
var bumpPtr int32 = 1024

//go:wasmexport alloc
func alloc(size int32) int32 {
	old := bumpPtr
	bumpPtr += size
	return old
}

//go:wasmexport free
func free(ptr int32, size int32) {
	// No-op: this is a bump allocator. We surrender the entire arena at
	// module teardown (the host disposes the wazero instance per call
	// when run_in_tx is true, so leaks are bounded by request duration).
	_ = ptr
	_ = size
}

//go:wasmexport auto_title
func auto_title(ptr int32, length int32) int64 {
	payload := readBytes(ptr, length)

	var req struct {
		Row struct {
			ID    string `json:"id"`
			Title string `json:"title"`
			Body  string `json:"body"`
		} `json:"row"`
	}
	if err := json.Unmarshal(payload, &req); err != nil {
		return errorEnvelope("invalid_payload", err.Error())
	}

	body := strings.TrimSpace(req.Row.Body)
	if body == "" {
		// Returning an error envelope with run_in_tx=true causes the
		// kernel to rollback the surrounding transaction — both the row
		// mutation and the action result are undone atomically.
		return errorEnvelope("body_required", "body is empty")
	}

	title := body
	if len(title) > 60 {
		title = title[:60]
	}
	// Trim trailing whitespace introduced by the cut.
	title = strings.TrimRight(title, " \t\n\r")

	return okEnvelope(map[string]any{
		"title": title,
	})
}

// ---- ABI helpers -----------------------------------------------------------

func okEnvelope(data any) int64 {
	body, _ := json.Marshal(map[string]any{
		"success": true,
		"data":    data,
		"meta":    map[string]any{"addon": "notes"},
	})
	return packResp(body)
}

func errorEnvelope(code, msg string) int64 {
	body, _ := json.Marshal(map[string]any{
		"success": false,
		"error":   map[string]any{"code": code, "message": msg},
		"meta":    map[string]any{"addon": "notes"},
	})
	return packResp(body)
}

func readBytes(ptr int32, length int32) []byte {
	// Unsafe pointer arithmetic is the standard trick for wasip1 modules
	// that expose `alloc` to the host: the host writes into guest memory
	// at the address we returned, so we re-read from the same address.
	src := unsafe.Slice((*byte)(unsafe.Pointer(uintptr(ptr))), int(length))
	out := make([]byte, len(src))
	copy(out, src)
	return out
}

func writePtr(b []byte) (int32, int32) {
	ptr := alloc(int32(len(b)))
	dst := unsafe.Slice((*byte)(unsafe.Pointer(uintptr(ptr))), len(b))
	copy(dst, b)
	return ptr, int32(len(b))
}

func packResp(b []byte) int64 {
	ptr, length := writePtr(b)
	return (int64(ptr) << 32) | int64(length)
}

// main is required by the wasip1 target. With -buildmode=c-shared it is
// never invoked (the runtime calls `_initialize` instead of `_start`).
func main() {}
```

Three things worth pausing on:

**Why packed `(ptr<<32) | len`?** WASM core 1 has no multi-value
returns, so the host-guest convention encodes a pointer + length pair
into one `i64`. The host unpacks it with the inverse shift — see
`kernel/runtime/wasm/abi.go:54-68` (host side) and
`packages/sdk/src/wasm-client.ts:261-271` (frontend side; same
encoding).

**Why `-buildmode=c-shared` and not the default?**

> If you compile without `-buildmode=c-shared` the module is a wasip1
> **command** that exports `_start` and calls `proc_exit(0)` as soon
> as `_start` returns. The host instantiates the module, the module
> immediately closes itself, and any subsequent `alloc` / handler
> call fails with `module is closed`.
>
> `-buildmode=c-shared` produces a wasip1 **reactor** instead: the
> module exports `_initialize` (called once at instantiation) and
> stays alive across host-driven calls. The two pilot addons learnt
> this the hard way during ops PR #132 — see the canonical
> incantation in `ops/addons-wasm/fiscal_mexico/build.sh:21-30`.

**Why a bump allocator?** It's two integer adds. The host disposes the
wazero instance per call when `run_in_tx: true` (one instance per
transaction), so the arena's lifetime is the request, not the addon.
Production addons that hold a long-lived module want a real free list;
the kernel doesn't care which.

### `backend/build.sh`

This is the canonical build script. Copy from `fiscal_mexico` and adjust
two strings:

```bash
#!/usr/bin/env bash
# Build the `notes` addon WASM module end-to-end:
#   1. Compile backend/main.go to backend/main.wasm (wasip1 reactor).
#   2. Regenerate signature.json with a sha256 digest over the wasm bytes.
set -euo pipefail

cd "$(dirname "$0")/.."

echo "==> Building backend/main.wasm (GOOS=wasip1 GOARCH=wasm)"
(
  cd backend
  # -buildmode=c-shared produces a wasip1 *reactor* (export _initialize)
  # rather than a command (export _start). Reactor semantics keep the
  # module alive across multiple host-driven calls — required for the
  # alloc + auto_title ABI. A command-style build would proc_exit(0)
  # after _start and close the module before the host can call exports.
  GOWORK=off GOOS=wasip1 GOARCH=wasm \
      go build -buildmode=c-shared -trimpath -o main.wasm .
)
ls -lh backend/main.wasm

# 2) Signature placeholder. The marketplace re-signs the bundle on
#    publish; this file just lets `addonbundle.VerifyBundle` recognise
#    the layout when ALLOW_UNSIGNED_BUNDLES is set in dev.
digest=$(sha256sum backend/main.wasm | awk '{print $1}')
signed_at=$(date -u +%Y-%m-%dT%H:%M:%SZ)
cat > signature.json <<EOF
{
  "developer_id": "you",
  "developer_name": "you",
  "verified": false,
  "signed_at": "${signed_at}",
  "algorithm": "sha256",
  "digest": "${digest}",
  "value": ""
}
EOF
echo "==> Wrote signature.json (digest=${digest})"
```

Compile it now:

```bash
bash backend/build.sh
# ==> Building backend/main.wasm (GOOS=wasip1 GOARCH=wasm)
# -rw-r--r-- 1 you you ~750K backend/main.wasm
# ==> Wrote signature.json (digest=...)
```

### `frontend/src/App.tsx`

The page uses primitives from `@asteby/metacore-ui` so it picks up the
host's theme without any extra wiring, and the metacore API client from
`@asteby/metacore-sdk` for the dynamic CRUD endpoints.

```tsx
import { useEffect, useState } from 'react'
import { Button, Card, Input, Textarea } from '@asteby/metacore-ui'

type Note = {
  id: string
  title: string
  body: string
  pinned: boolean
}

/**
 * Notes list page — federated remote exposed as `./App` in vite.config.
 *
 * Data flows through the kernel's dynamic CRUD endpoints:
 *   GET    /api/dynamic/notes_note               → list
 *   POST   /api/dynamic/notes_note               → create
 *   PUT    /api/dynamic/notes_note/:id           → update
 *   DELETE /api/dynamic/notes_note/:id           → delete
 *   POST   /api/dynamic/notes_note/:id/action/auto_title
 *
 * Endpoint shape: `kernel/dynamic/handler.go:32-37`.
 * Response envelope `{success, data, meta}` is the kernel-wide convention.
 */
export default function App() {
  const [notes, setNotes] = useState<Note[]>([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    void refresh()
  }, [])

  async function refresh() {
    setLoading(true)
    const resp = await fetch('/api/dynamic/notes_note', {
      credentials: 'include',
    })
    const env = await resp.json()
    setNotes(env.data ?? [])
    setLoading(false)
  }

  async function create() {
    await fetch('/api/dynamic/notes_note', {
      method: 'POST',
      credentials: 'include',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({
        title: 'Untitled',
        body: '',
        pinned: false,
      }),
    })
    await refresh()
  }

  async function update(id: string, patch: Partial<Note>) {
    await fetch(`/api/dynamic/notes_note/${id}`, {
      method: 'PUT',
      credentials: 'include',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify(patch),
    })
    await refresh()
  }

  async function autoTitle(id: string) {
    // POST /dynamic/:model/:id/action/:key — see kernel/dynamic/handler.go:37
    // and the ActionTrigger contract in kernel/manifest/manifest.go:252.
    await fetch(`/api/dynamic/notes_note/${id}/action/auto_title`, {
      method: 'POST',
      credentials: 'include',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({}),
    })
    await refresh()
  }

  async function destroy(id: string) {
    await fetch(`/api/dynamic/notes_note/${id}`, {
      method: 'DELETE',
      credentials: 'include',
    })
    await refresh()
  }

  if (loading) return <div className="p-4">Loading…</div>

  return (
    <div className="p-4 space-y-4">
      <div className="flex items-center justify-between">
        <h1 className="text-xl font-semibold">Notes</h1>
        <Button onClick={create}>New note</Button>
      </div>

      <div className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 gap-4">
        {notes.map((n) => (
          <Card key={n.id} className="p-4 space-y-2">
            <Input
              value={n.title}
              placeholder="Title"
              onChange={(e) => update(n.id, { title: e.target.value })}
            />
            <Textarea
              value={n.body}
              placeholder="Write something…"
              rows={6}
              onChange={(e) => update(n.id, { body: e.target.value })}
            />
            <div className="flex gap-2">
              <Button variant="secondary" onClick={() => autoTitle(n.id)}>
                Auto title
              </Button>
              <Button variant="ghost" onClick={() => destroy(n.id)}>
                Delete
              </Button>
            </div>
          </Card>
        ))}
      </div>
    </div>
  )
}
```

Two things worth noting:

- `credentials: 'include'` is required so the kernel's session cookie
  reaches the dynamic handler. The host's API client (`@asteby/metacore-sdk`'s
  `createApiClient`) does this for you — we inline `fetch` here only to
  keep the example self-contained.
- The created note's UUID is returned by the kernel and used as the row
  key. The `Card`'s inputs are uncontrolled-ish — each keystroke fires
  a PUT. A production addon would debounce; we don't, on purpose, so
  the demo stays under 100 lines.

```

## Step 5 — Install the addon

The kernel exposes a marketplace install endpoint:

```

## Step 6 — Test the action

Open the host, navigate to **Notes** in the sidebar, click **New note**.
Type a paragraph into the body of one card and click **Auto title**.

What happens server-side:

1. The browser POSTs to
   `/api/dynamic/notes_note/<id>/action/auto_title`.
2. The kernel's dynamic action handler resolves the ActionDef, sees
   `trigger.type: "wasm"` and dispatches to the addon's wazero
   instance.
3. Because `run_in_tx: true`, the dispatch happens inside the same DB
   transaction as the row read. The host marshals the row into the
   handler's JSON envelope (`{"row": {"id": "...", "title": "...",
   "body": "..."}}`), calls `auto_title(ptr, len)`, unpacks the
   returned slot.
4. The handler returns `{"success": true, "data": {"title": "..."}}`.
   The kernel applies the patch and commits.
5. If the handler instead returns `{"success": false, ...}` — try it
   by emptying the body before clicking the action — the kernel
   rolls the transaction back. No partial state ever lands in the DB.
   The same rollback path is exercised by `ops` PR #132's integration
   test (`fiscal_mexico` returns an error envelope when `rfc_emisor`
   is empty and asserts the row is unchanged).

The action emits no canonical event by itself; if you want one, add
`fiscal.invoice.stamped`-style entries to `manifest.events[]` and call
the `event_emit` host import from your WASM module. The contract is at
`kernel/runtime/wasm/abi.go:21-26` (host imports) and was wired into the
events.Bus in kernel PR #54.

## Step 7 — Hot-swap

Change `auto_title` in `backend/main.go` — for instance, trim to 80
characters instead of 60:

```diff
-       if len(title) > 60 {
-               title = title[:60]
-       }
+       if len(title) > 80 {
+               title = title[:80]
+       }
```

Rebuild and re-drop:

```bash
bash backend/build.sh
cp backend/main.wasm signature.json /path/to/host/addons/notes/backend/
```

What happens, without restarting the host:

1. The kernel's installer recomputes the manifest hash, sees it
   changed, and emits an `ADDON_MANIFEST_CHANGED` message over the
   `ws.Hub` (kernel PR #52 added the broadcaster, PR #54 wired it).
2. The frontend's `wireHotSwapInvalidation` subscriber
   (`packages/runtime-react/src/manifest-hotswap-subscriber.ts`) sees
   the message and runs the `applyHotSwapReload` action configured by
   the host — `rekey` (re-mount the addon route), `page-reload`
   (full refresh) or `manual` (raise an event the host UI handles).
3. The metadata cache is invalidated so any subsequent SDK call
   refetches the manifest.

The user sees the new behaviour on the next interaction — no host
restart, no F5. Strategy details and the `useHotSwapReload` hook are at
[`packages/runtime-react/src/hotswap-reload-policy.ts`](../../packages/runtime-react/src/hotswap-reload-policy.ts).

> **Frontend changes need a fresh remoteEntry.js** — the kernel's
> hot-swap path covers manifest + wasm. If you change `App.tsx` you need
> to `pnpm build` the frontend again; the host's federation loader will
> pick the new bundle up on the next route mount (or immediately, with
> the `rekey` strategy and a cache-busted `?v=<hash>` query).

## Step 8 — Immersive layout (optional)

Some addons own the whole viewport: a kitchen display, a POS, a
waiting-room screen. Set `frontend.layout` to `"immersive"`:

```diff
-    "layout":    "shell"
+    "layout":    "immersive"
```

Reinstall (drop + reload). The host now mounts `<App />` without the
sidebar / topbar; routing back is your problem (a back button, a
keyboard shortcut, the browser's history). The contract is at
`kernel/manifest/manifest.go:125-141` (`FrontendSpec.Layout`); the
reference implementation is `ops/addons-wasm/kitchen-display` which
runs in immersive mode by default.

`"shell"` is the implicit default for legacy manifests that never set
the field, so existing addons keep their chrome.
